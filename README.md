# AI 实时对话画布

基于 Vite + React + TypeScript + Tailwind CSS v4 构建的纯前端 AI 对话画布应用。用户通过自然语言指令在 Canvas 上绘制矢量图形，支持 undo/redo 和历史指令回滚。

## 本地启动方式

```bash
# 安装依赖
npm install

# 启动开发服务器
npm run dev

# 构建生产版本
npm run build

# 预览生产版本
npm run preview
```

启动后访问 `http://localhost:5173` 即可使用。

## 功能特性

- ✅ 自然语言指令解析（中英文混合支持）
- ✅ Canvas 矢量图形增量渲染
- ✅ 每次新指令不清除已有图形
- ✅ Undo / Redo 操作（支持快捷键 Ctrl+Z / Ctrl+Y）
- ✅ 历史指令列表，点击任意指令可回滚到该状态
- ✅ 内置规则引擎，无需接入真实 LLM
- ✅ 多端协同绘图（基于 BroadcastChannel API，无需后端）
- ✅ 实时同步延迟 < 200ms
- ✅ 历史列表与 undo/redo 栈双向同步一致
- ✅ 连接状态实时显示（已连接/未连接/同步中）
- ✅ 防循环同步机制

## 协同模式

### 技术原理

基于浏览器原生通信 API 实现同源跨标签页协同绘图，无需后端服务器。

- **主通道**: `BroadcastChannel API`（优先使用，低延迟、高效率）
- **降级通道**: `localStorage + storage 事件`（BroadcastChannel 不可用时自动启用）
- **消息类型**: 指令添加、撤销、重做、清空、回滚、状态同步、心跳、握手
- **防循环机制**:
  1. 每条消息携带唯一 `senderId`，接收方忽略自己发送的消息
  2. 每条消息携带单调递增 `version`，接收方仅处理版本号更新的消息
  3. localStorage 通道附加 `nonce` 去重，避免重复消费
- **状态一致性**: 使用 `commands` 数组 + `currentIndex` 指针模型，所有操作都是确定性的，两端执行相同操作序列即可保证状态一致
- **新客户端同步**:
  1. 新连接客户端发送 `hello` 握手消息
  2. 已有客户端若有内容则主动推送 `state-sync` 全量状态
  3. 新客户端同时发送 `state-request` 请求状态
  4. **只有在收到 `state-sync` 或确认双方已对齐后，才从「同步中」转为「已连接」**
  5. 设置 3 秒超时兜底：若超时仍未收到状态同步但已检测到 peers，则降级为已连接

### 同源限制与 Fallback 机制

> ⚠️ **重要**：无论是 BroadcastChannel 还是 localStorage，都受浏览器同源策略限制。
>
> **Origin 必须完全一致**才能通信，以下组合**无法协同**：
> - `http://localhost:5173` vs `http://127.0.0.1:5173`（主机名不同）
> - `http://localhost:5173` vs `http://localhost:3000`（端口不同）
> - `http://localhost:5173` vs `https://localhost:5173`（协议不同）

**Fallback 策略**：
1. 优先尝试 `BroadcastChannel`（延迟 < 10ms）
2. 若 BroadcastChannel 创建失败，自动降级到 `localStorage + storage 事件`（延迟 < 50ms）
3. 两者都不可用时，状态栏显示「未连接」并提示检查 origin
4. 工具栏显示当前使用的传输通道（BroadcastChannel / LocalStorage）和当前 origin

### 连接状态说明

工具栏显示协同连接状态：

| 状态 | 指示灯颜色 | 说明 |
|-----|-----------|------|
| 🟢 已连接 | 绿色 | 已完成状态同步，检测到其他在线客户端 |
| ⚪ 未连接 | 灰色 | 未检测到其他在线客户端，或 origin 不一致 |
| 🟡 同步中 | 黄色 + 脉冲动画 | 已发现 peer，正在进行全量状态同步 |

**关键改进**：新客户端连接时会先进入「同步中」状态，**只有完成状态同步后才显示「已连接」**，避免用户看到"已连接"但画布为空的误导性状态。

已连接状态下还会显示：
- 在线客户端数量（指示灯右上角角标）
- 同步延迟（单位 ms，应 < 200ms）
- 传输通道类型（BroadcastChannel / LocalStorage）
- 当前同步版本号
- 悬停可查看当前 origin

未连接状态下会显示：
- 当前 origin 提示，帮助用户排查跨 origin 问题

### 使用说明

1. **启动协同会话**
   - 启动开发服务器：`npm run dev`
   - **注意所有协同端必须使用完全相同的 URL（包括协议、主机名、端口）**
   - 推荐统一使用 `http://localhost:5173`
   - 在同一浏览器中打开第二个标签页，访问相同地址
   - 初始状态为「同步中」，完成状态同步后自动变为「已连接」

2. **协同绘图**
   - 在任意一个标签页发送绘图指令
   - 另一个标签页会实时同步显示相同的图形
   - 两端的历史指令列表保持完全一致

3. **协同 Undo/Redo**
   - 在任意一端点击 Undo 或按 Ctrl+Z
   - 另一端同步撤销最后一条指令
   - Redo 操作同理

4. **协同清空**
   - 在任意一端点击「清空」按钮
   - 另一端同步清空所有指令和画布

5. **历史回滚同步**
   - 在任意一端点击历史指令列表中的任意条目
   - 另一端同步回滚到该状态

6. **排查连接问题**
   - 若长时间显示「未连接」，检查两端 URL 的 origin 是否完全一致
   - 状态栏会显示当前 origin，对比两者是否相同
   - 不要混用 `localhost` 和 `127.0.0.1`

## 验证方式

### 基础功能验证

1. 启动应用：`npm run dev`
2. 依次发送以下三条指令：
   ```
   画一个红色圆形在左上角
   画一个蓝色矩形在右下角
   画一个绿色三角形在中间
   ```
3. 观察画布：应正确累积显示三个不同的图形
4. 点击 Undo 按钮（或按 Ctrl+Z）：最后一个绿色三角形应被撤销
5. 点击 Redo 按钮（或按 Ctrl+Y）：绿色三角形应重新出现

### 协同功能验证

#### 1. 连接状态验证
1. 启动应用：`npm run dev`
2. 在浏览器中打开第一个标签页访问 `http://localhost:5173`
3. 观察工具栏：显示「未连接」状态
4. 在同一浏览器中打开第二个标签页访问同一地址
5. 观察两个标签页的工具栏：均应在 1 秒内变为「已连接」状态，显示在线客户端数量为 1

#### 2. 绘图同步验证
1. 在第一个标签页发送指令：`画一个红色圆形在左上角`
2. 观察第二个标签页：应在 200ms 内显示相同的红色圆形，历史列表同步更新
3. 在第二个标签页发送指令：`画一个蓝色矩形在右下角`
4. 观察第一个标签页：应在 200ms 内显示相同的蓝色矩形，历史列表同步更新
5. 验证两端历史列表完全一致，指令顺序相同

#### 3. Undo/Redo 同步验证
1. 在第一个标签页点击 Undo 按钮
2. 观察第二个标签页：同步撤销最后一条指令，蓝色矩形消失
3. 在第二个标签页点击 Redo 按钮
4. 观察第一个标签页：同步重做，蓝色矩形重新出现
5. 验证两端 Undo/Redo 按钮状态完全一致

#### 4. 历史回滚同步验证
1. 在第一个标签页点击历史列表中的第一条指令
2. 观察第二个标签页：同步回滚到该状态，仅显示红色圆形
3. 验证两端 currentIndex 指针位置一致

#### 5. 清空同步验证
1. 在第一个标签页点击「清空」按钮
2. 观察第二个标签页：同步清空所有指令，画布变为空白
3. 验证两端历史列表均为空，Undo/Redo 按钮均为禁用状态

#### 6. 新客户端全量同步验证
1. 保持前两个标签页打开，且已有若干绘图指令
2. 打开第三个标签页访问同一地址
3. 观察第三个标签页：应显示「同步中」状态，然后自动同步所有历史指令和画布内容
4. 验证三个标签页的状态完全一致

#### 7. 延迟验证
1. 连续发送多条绘图指令
2. 观察工具栏显示的同步延迟
3. 验证延迟始终低于 200ms（通常在 10ms 以内）

#### 8. 防循环同步验证
1. 在任意一端发送绘图指令
2. 观察另一端同步后，不会再次触发消息发送
3. 持续操作 1 分钟，验证没有出现指令重复、死循环等问题
